Go编程规范

Written by Inno with ♥ on February 11, 2020 in golang Total 3263 words

要使用好你的语言，重要的是要了解它的特性和约定。 了解Go编程中已建立的约定（如命名，格式设置，程序构造等）非常重要，这样您编写的程序将易于其他Go程序员理解。以下我们介绍了惯用的Go代码的技巧和语言规范。

注释

Go提供了C样式/ * * /块注释和C ++样式//行注释。 行注释是常态。 块注释主要显示为程序包注释，但在表达式中很有用，或者用于禁用大量代码。

GO程序和Web服务器通过Godoc处理Go源文件以提取有关软件包内容的文档。 在顶级声明之前出现的注释（没有中间的换行符）与声明一起被提取，以用作该项目的解释性文本。 这些注释的性质和样式决定了godoc生成的文档的质量。

每个包都应在包子句之前有一个包注释（一个块注释）。 对于多文件包，包注释仅需要出现在一个文件中，任何一个都可以。 包装评论应介绍包装，并提供与包装整体有关的信息。 它会首先出现在godoc页面上，并应设置随后的详细文档。

当然如果包很简单，那么包注释可以很简短。

Package path implements utility routines for

在包中，顶级声明之前的任何注释都将用作该声明的doc注释。 程序中的每个导出（大写）名称都应带有文档注释。

非顶级声明注释最好作为完整的句子使用，简述倒出元素作用，第一句应为单句摘要，以声明的名称开头。

// Compile parses a regular expression and returns, if successful,
// a Regexp that can be used to match against text.
func Compile(str string) (*Regexp, error) {

在注释中我们可以还可以使用tab的方式标注代码段。

/*
数据结构-图
	type GRAPH
*/

命名规范和原则

在GO语言中，名字和其他语言一样重要。它们甚至具有语义效应：包外部名称的可见性取决于它的第一个字符是否为大写。因此，花点时间讨论一下GO语言程序中的命名约定是很重要的的。

File names

由于文件跟包无任何关系， 而又避免windows大小写的问题，所以推荐的明明规范如下： 文件名应一律使用小写， 不同单词之间用下划线分割, 命名应尽可能地见名知意

Package names

当我们导入一个包时，其包名成为其内容的访问器，就好比下面我们倒入byte包。

import "byte"

包中的bytes.Buffer就是一个很好的例子。因为每个使用这个包的人都使用相同的包名来引用它的内容，这意味着这个包的名称应该是简短、简明、能引起共鸣的。按照惯例，包被给予小写的，单字(不要使用复合名词)的名字，不需要下划线或混合大写。包名只是导入的默认名称，它不需要在所有源代码中都是唯一的，并且在发生冲突的罕见情况下，导入包可以选择在本地使用不同的名称。在任何情况下，都很少出现包内容的混淆，因为导入中的文件名只决定使用哪个包。

另一个约定是，程序包名称是其源目录(文件夹)的名称。 src/encoding/base64中的包被导入为“encoding / base64”，但名称为base64，而不是encoding_base64，也不是encodingBase64。

包的导入者将使用该包的名称来引用其内容，因此，包中导出的名称可以避免冲突。例如，**bufio包中的缓冲读取器类型称为Reader，而不是BufReader，**因为用户将其视为bufio.Reader，这是一个简洁明了的名称。 此外，由于导入的实体始终使用其包名称来寻址，因此bufio.Reader不会与io.Reader冲突。 同样，用于创建ring的新实例的函数。Ring（这是Go中构造函数的定义）通常被称为NewRing，但是如果Ring是该包导出的唯一类型，并且由于该包被称为ring，因此Ring的构造函数称为New，使用包结构可以帮助您选择好名字，当然这局限于包中需要暴露出一个构造函数。

Getters

Go不为getter和setter提供支持。 自己提供getter和setter并没有错，这样做通常是适当的，但是也要遵循一些规则。

如果您有一个名为owner（小写，未导出）的字段，则getter方法应称为Owner（大写，导出），而不是GetOwner。 使用大写名称进行导出可提供钩子，以将字段与方法区分开。 如果需要的话，一个setter函数将被称为SetOwner。

owner := obj.Owner()
if owner != user {
    obj.SetOwner(user)
}

Interface names

按照惯例，一个方法接口的名称是由方法名加上一个**-er**后缀或类似的修改来构造一个代理名词：Reader、Writer、Formatter等。

有很多这样的名字：读、写、关闭、刷新、字符串等等都具有规范的签名和含义。为了避免混淆，不要给你的方法起这些名字，除非它们有相同的签名和含义。相反，如果您的类型实现了与已知类型上的方法具有相同含义的方法，则为其提供相同的名称和签名比如调用字符串转换器方法String而不是ToString。

MixedCaps

最后大小写混合，Go中的惯例是使用MixedCaps/mixedCaps(根据可访问性来决定)，而不是下划线来编写多字名称。

总结

  • 包名使用小写单字母。

  • 包中的对外实体结合包的名称带来的语意来命名。

  • 当包只暴露一个实例化函数的时候使用New作为其方法名。

  • Getter方法直接首字母大写要访问的字段名，Setter方法在要访问的字段前面加Set。

  • 接口以er结尾来构造代理的含义。

  • 使用大小写混合，而不是下划线。

分号

与C语言一样，Go的正式语法使用分号来结束语句，但与C语言不同，这些分号不会出现在源代码中。相反，词法分析器使用一个简单的规则在扫描时自动插入分号，因此输入文本基本上没有分号。

规则是这样的。如果换行之前的最后一个标记是一个标识符(其中包括int和float64这样的单词)，那么它就是一个基本的文字，。比如数字或字符串常量，或者以下一个标记

break continue fallthrough return ++ -- ) } ] :

词法分析器总是在标记后面插入一个分号。这可以总结为，如果换行符位于可以结束语句的标记之后，则插入一个分号。分号也可以直接省略在右大括号之前，因此语句如

go func() { for { dst <- <-src } }()

分号插入规则的一个后果是不能将控制结构(if、for、switch或select)的左大括号放在下一行。如果这样做，将在大括号前插入分号，这可能会造成不必要的影响。像这样写就会爆出不必要的错误。

if i < f()  // wrong!
{           // wrong!
    g()
}

编程风格

变量分组

将函数要使用的变量分组并且在一开始声明，这样的声明方式更加的清晰，将函数赋值和错误处理写在一行，避免代码的冗杂。

func (c *Connection) readLoop() {
	var (
		req         *Message
		messageType int
		data        []byte
		err         error
	)
	for {
		if messageType, data, err = c.conn.ReadMessage(); err != nil {
			goto error
		}
		req = &Message{
			messageType: messageType,
			data:        data,
		}
		select {
		case c.inChan <- req:
		case <-c.closeChan:
			goto closed
		}
	}
error:
	c.close()
closed:
}

模块初始化

将不同的模块在mian函数里做初始化。

package main

func run(topic string) {
	//......
}

func main() {
  // 初始化kafka
	err = kafka.Init([]string{cfg.Address}...)
	if err != nil {
		fmt.Printf("Init kafka fail, err:%v",err)
		return
	}
	fmt.Println("kafka load success")
	// 打开日志文件收集日志。
	err = tailLog.Init(cfg.FilePath)
	if err != nil {
		fmt.Printf("Init log tail fail, err:%v",err)
		return
	}
	fmt.Println("tail log load success")
	run(cfg.Topic)
}

格式化

Golang的开发团队制定了统一的官方代码风格，并且推出了gofmt工具（gofmt或go fmt）来帮助开发者格式化他们的代码到统一的风格。gofmt是一个cli程序，会优先读取标准输入，如果传入了文件路径的话，会格式化这个文件，如果传入一个目录，会格式化目录中所有.go文件，如果不传参数，会格式化当前目录下的所有.go文件。

gofmt默认不对代码进行简化，使用-s参数可以开启简化代码功能，具体来说会进行如下的转换：

  • 去除数组、切片、Map初始化时不必要的类型声明：
如下形式的切片表达式：
    []T{T{}, T{}}
将被简化为：
    []T{{}, {}}
  • 去除数组切片操作时不必要的索引指定
如下形式的切片表达式：
    s[a:len(s)]
将被简化为：
    s[a:]
  • 去除迭代时非必要的变量赋值
如下形式的迭代：
    for x, _ = range v {...}
将被简化为：
    for x = range v {...}
如下形式的迭代：
    for _ = range v {...}
将被简化为：
    for range v {...}

gofmt命令参数列表如下：

usage: gofmt [flags] [path ...]
  -cpuprofile string
        write cpu profile to this file
  -d    display diffs instead of rewriting files
  -e    report all errors (not just the first 10 on different lines)
  -l    list files whose formatting differs from gofmt's
  -r 		string	rewrite rule (e.g., 'a[b:len(a)] -> a[b:]')
  -s    simplify code
  -w    write result to (source) file instead of stdout

goland配置

在Goland中，可以通过添加一个File Watcher来在文件发生变化的时候调用gofmt进行代码格式化，具体方法是，点击Preferences -> Tools -> File Watchers，点加号添加一个go fmt模版，Goland中预置的go fmt模版使用的是go fmt命令，将其替换为gofmt，然后在参数中增加-l -w -s参数，启用代码简化功能。添加配置后，保存源码时，goland就会执行代码格式化了。
